## 1. 프로젝트 한 줄 정의

> 대학생의 과제·시험·일정을 AI가 분석하여 매일 최적의 행동을 추천해주는 **개인 운영 비서 앱**

---

## 2. 배경과 동기

대학생은 매 학기 수십 개의 과제, 시험, 팀플을 동시에 관리해야 한다.  
현실은 이렇다:

- 과제 공지가 LMS, 카톡, 구두 등 **여러 채널에 분산**되어 놓치기 쉽다.
- 마감이 겹치면 **뭘 먼저 해야 할지 판단을 못 하고** 쉬운 것만 먼저 한다.
- 매일 아침 "오늘 뭐 하지?"를 고민하는 데 **20분씩 낭비**한다.
- Notion, Todoist 같은 도구는 결국 **내가 직접 정리해야** 한다.

**기록은 충분하다. 부족한 건 판단이다.**

---

## 3. 핵심 문제 정의

### 우리가 해결하는 문제

**"대학생은 할 일이 많아서 힘든 게 아니라, 뭘 먼저 해야 할지 몰라서 힘들다."**

### 기존 해결책의 한계

| 도구 | 하는 일 | 못하는 일 |
|---|---|---|
| **Notion** | 자유롭게 기록·정리 | 우선순위 판단, 행동 추천 |
| **Todoist** | 할 일 체크리스트 관리 | 마감·난이도 기반 자동 정렬 |
| **Google Calendar** | 시간 블록 시각화 | "이 시간에 뭘 해야 하는지" 제안 |
| **에브리타임** | 시간표·커뮤니티 | 과제/시험 관리 기능 없음 |

### 우리의 차별점

**"기록하는 앱"이 아니라 "판단해주는 앱"**

- 사용자는 과제를 **던져넣기만** 하면 된다 (텍스트든 캡처든).
- AI가 마감일, 난이도, 시험 충돌, 소요시간을 **종합 분석**한다.
- 매일 "오늘 할 일"을 **구체적으로 추천**해준다.

---

## 4. 타겟 사용자

### 1차 타겟

- **전공 과제가 많은 3~4학년 대학생**
- 한 학기에 과제 10개 이상, 시험 5회 이상
- "체계적으로 하고 싶은데 방법을 모르겠는" 유형

### 비타겟 (지금은 노리지 않는 사용자)

- 과제가 적은 1~2학년 교양 위주 수강생
- 이미 Notion으로 완벽하게 관리하고 있는 사람
- 대학원생, 직장인 (학부생에 집중)

## 5. 핵심 컨셉

### 설계 원칙 3가지

| 원칙 | 설명 |
|---|---|
| **제로 정리** | 사용자가 정리할 필요 없다. 던져넣으면 AI가 구조화한다. |
| **매일 한 문장** | "오늘 2시간은 DB 과제" — 복잡한 대시보드보다 한 줄 추천이 핵심이다. |
| **틀려도 괜찮다** | AI 추천이 100% 정확할 필요 없다. "아, 맞다 그거 해야지"를 떠올리게 하면 성공이다. |

### 사용자 시나리오

```
1. 수업 끝나고 교수님이 과제 공지 → 캡처해서 앱에 업로드
2. AI가 "DB설계 과제 / 마감 6월 10일 / 예상 5시간"으로 자동 파싱
3. 난이도를 "상"으로 수정하고 저장
4. 다음 날 아침 앱 열면:
   "오늘 추천: DB 과제 2시간 (마감 D-5, 우선순위 1위) + 영어 에세이 초안 1시간"
5. 과제 완료 후 체크 → 다음 추천 자동 갱신
```

## 6. MVP 기능 범위

### 포함하는 것

| 기능 | 상세 | 우선순위 |
|---|---|---|
| **과제 입력** | 텍스트 직접 입력 + 이미지 업로드(캡처) → AI가 과제명·마감일·과목 자동 추출 | P0 |
| **AI 우선순위 계산** | 마감 잔여일, 난이도(사용자 입력), 시험 충돌 여부, 예상 소요시간을 종합해 우선순위 스코어 산출 | P0 |
| **오늘 할 일 추천** | 가용 시간 입력 → 시간 블록별 과제 배분 추천 | P0 |
| **과제 대시보드** | 전체 과제 목록, 마감 임박 하이라이트, 완료 체크 | P1 |
| **사용자 인증** | 이메일 또는 소셜 로그인 | P1 |

### 의도적으로 제외하는 것

| 제외 기능 | 제외 이유 |
|---|---|
| 팀플 관리 | 멀티유저 로직은 복잡도가 급증, MVP에서는 개인 사용에 집중 |
| PDF 요약 | 과제 관리와 별개 기능, 핵심 가치와 직결되지 않음 |
| 시험기간 모드 | 우선순위 로직이 안정화된 후에 추가해야 의미 있음 |
| 캘린더 연동 | OAuth 연동 복잡도 대비 MVP 검증에 필수가 아님 |
| 네이티브 앱 | 웹 반응형으로 모바일 접근성 확보, 검증 후 앱 전환 |
| 푸시 알림 | MVP에서는 앱 접속 시 추천으로 충분 |

> **제외 판단 기준**: "이 기능이 없으면 핵심 가치(AI 판단·추천)를 검증할 수 없는가?" → 아니면 제외.

## 7. 기술 스택과 선택 이유

| 레이어 | 기술 | 왜 이걸 선택했는가 |
|---|---|---|
| **프론트엔드** | Next.js (App Router) | SSR로 초기 로딩 빠름, API Route로 백엔드 로직 통합 가능, 1인 개발 시 프론트·백 분리 부담 최소화 |
| **백엔드/DB** | Supabase (PostgreSQL) | Auth·Storage·DB가 한 곳에 통합, SQL 기반이라 관계형 데이터(과제-과목-시험) 표현에 적합, Free Tier로 MVP 비용 제로 |
| **AI** | OpenAI API (GPT-4o) | 자연어 이해력 최상위, Vision API로 이미지 파싱까지 하나로 해결, Function Calling으로 정형 JSON 응답 추출 가능 |
| **배포** | Vercel | Next.js 공식 배포 플랫폼, PR마다 Preview Deploy 자동 생성, 무료 Hobby 플랜으로 충분 |
| **스타일링** | Tailwind CSS + shadcn/ui | 빠른 UI 개발 + 일관된 디자인 시스템, 커스터마이징 자유도 높음 |

## 8. 성공 기준

### "이 프로젝트가 잘 됐다"의 기준

**기능 검증 (6주 후)**

- [ ] 과제 입력 → 우선순위 산출 → 오늘 추천까지 **End-to-End로 동작**한다.
- [ ] AI 추천 결과의 **70% 이상**이 스스로 "납득 가능하다"고 느낀다.
- [ ] AI 추천 응답이 **3초 이내**에 돌아온다.
- [ ] **본인이 2주 연속 매일 1회 이상** 실제로 사용한다.

**사용자 검증 (8주 후)**

- [ ] 주변 동기/후배 **5명 이상**이 베타 테스트에 참여한다.
- [ ] 5명 중 **3명 이상**이 1주일 후에도 접속한다.
- [ ] "친구한테 추천하겠다"고 답한 사람이 **3명 이상**이다.

> **핵심 판단**: 내가 2주 동안 매일 쓰면서 "이거 없으면 불편하다"고 느끼면 성공이다.

## 9. 리스크와 대응

| # | 리스크 | 왜 위험한가 | 대응 방법 |
|---|---|---|---|
| 1 | **AI 추천 품질이 낮음** | 추천이 엉뚱하면 앱의 존재 이유가 없어진다 | 프롬프트 버전 관리, Few-shot 예시 10개 이상 축적, 사용자 피드백("별로야") 반영 루프 |
| 2 | **OpenAI API 비용 폭증** | 사용자 늘면 비용이 선형 증가 | 요청당 토큰 제한, 동일 과제 재분석 캐싱, GPT-4o-mini 폴백 전략 |
| 3 | **이미지 파싱 부정확** | 캡처 품질에 따라 추출 실패 가능 | 파싱 결과를 사용자가 확인·수정하는 UI 제공, 텍스트 입력을 기본 경로로 유지 |
| 4 | **1인 개발 번아웃** | 기획·개발·디자인·테스트 전부 혼자 | 주 단위 마일스톤으로 작게 쪼개기, "동작하면 OK" 마인드, 주 1회 5분 회고 |
| 5 | **베타 사용자 무관심** | 주변에 깔아달라고 해도 안 씀 | MVP 전에 주변 3명에게 시연, 피드백 기반으로 "이게 정말 필요한가" 검증 선행 |

> **리스크 #1이 가장 치명적이다.** 개발 초기에 프롬프트 실험에 충분한 시간을 투자해야 한다.

## 10. 향후 확장 방향

MVP 검증이 끝난 후, 아래 순서로 확장한다.

### 2단계 — 학습 지원 강화

- **시험기간 모드**: 시험 일정 입력 → 복습 스케줄 자동 생성, 과제 우선순위 재조정
- **PDF 요약**: 강의자료 업로드 → 핵심 요약 + 예상 문제 생성
- **팀플 관리**: 팀원 초대, 역할 분담, 진행률 공유

### 3단계 — AI 고도화

- **발표 피드백**: 발표 스크립트 → 논리 흐름·시간 배분 AI 코칭
- **실습일지 자동화**: 실습 메모 → 정형화된 보고서 자동 생성
- **스트레스 기반 조절**: 사용자 컨디션 입력 → 일정 완화/집중 모드 전환

> 확장은 MVP에서 **"이 앱을 매일 쓴다"가 증명된 후**에만 진행한다.
