## priority.ts

### 잘 된 점

- `deadlineBucket + score` 2단계 정렬은 직관적이고 납득 가능하다. 순수 점수 정렬과 달리 "내일 마감 쉬운 과제"가 "7일 뒤 어려운 과제"보다 먼저 나온다.
- `scheduled / overflow / overdue` 3분류는 관심사 분리가 명확하다. overdue를 배정 루프 밖으로 빼서 점수 계산에 영향을 주지 않는 것이 특히 좋다.
- `buildPrioritySummary`를 active 과제 전용으로 좁힌 것이 올바른 방향이다.
- `remainingHours` 필드 추가로 UI에서 인라인 계산 없이 타입 안전하게 남은 시간을 참조할 수 있다.

### 개선 포인트

#### [P2] `differenceInDays` 이중 호출

```typescript
// 현재: 각 과제마다 2번 호출 (filter 1번 + map 1번)

const activeRaw = nonDone.filter(
  (a) => differenceInDays(a.dueDate, today) >= 0

```

`map` 한 번으로 `daysLeft`를 먼저 계산하고 분기하는 게 더 효율적이다.

```typescript
// 개선 방향
const allScored = nonDone.map((a) => ({ ...a, daysLeft: differenceInDays(a.dueDate, today) }));
const overdueRaw = allScored.filter((a) => a.daysLeft < 0);
const activeRaw  = allScored.filter((a) => a.daysLeft >= 0);
```

`ScoredAssignment` 주석에 `99=마감초과`라고 적어두긴 했지만, overdue 항목은 애초에 `deadlineBucket`을 통하지 않으므로 named constant가 더 명시적이다.

#### [P3] `partialOverflow` 중복 노출 의도 명시 필요

부분 배정된 과제는 `scheduled`(오늘 카드)와 `overflow`(내일 플랜) 양쪽에 모두 들어간다. 이것은 의도된 설계지만 코드만 보면 버그처럼 보일 수 있다. 주석으로 명시하는 것을 권장한다.


## TodayPlanner.tsx

### 잘 된 점

- `useRef`를 이용한 AI 요청 race condition 처리가 정석적이다. `reqId` 캡처 후 응답 도착 시 `aiRequestId.current === reqId` 비교로 stale 응답을 정확히 걸러낸다.
- `handleComplete`에서 `aiLoading(false)` 즉시 호출로 스피너가 남는 문제를 해결했다. `aiRequestId.current++`와 함께 호출하는 위치가 정확하다.
- `OverduePlan`을 별도 컴포넌트로 분리해 주황색 테마로 구분한 것이 UX상 적절하다.

### 개선 포인트

#### [P2] `calcBlockedHours` 매 렌더마다 재실행

현재 `blockedHours`는 렌더 함수 안에서 바로 계산된다. `schedules` prop이 바뀌지 않아도 다른 state 변경(hours 입력, completingId 등)으로 리렌더가 일어날 때마다 재실행된다. `useMemo`로 감싸는 게 맞다.

```typescript
const blockedHours = useMemo(
  () => calcBlockedHours(schedules),
  [schedules]
);
```

#### [P2] `handleComplete`의 연속 완료 처리

현재 `handleComplete`는 `assignments` prop을 기준으로 필터링한다. A를 완료하고 B를 완료하는 경우, B를 완료할 때 `recommend()`에 넘기는 배열에는 방금 완료한 A도 없어야 하는데, `assignments` prop은 `router.refresh()` 완료 전까지 갱신되지 않으므로 A가 다시 포함될 수 있다.

실제 사용에서는 A 완료 직후 B 완료 버튼을 누르기 전에 서버 응답이 오는 경우가 대부분이라 심각한 버그는 아니다. 단, 빠른 연속 완료에 대비하려면 별도의 `completedIds` ref로 누적 필터링이 필요하다.

```typescript
// 개선 방향 (필요 시)
const completedIds = useRef<Set<string>>(new Set());

function handleComplete(id: string) {
  completedIds.current.add(id);
  // ...
  const remaining = assignments.filter(
    (a) => !completedIds.current.has(a.id) && a.status !== "done"
  );
}
```

## ai.ts

### 잘 된 점

- 시스템 프롬프트를 영어로 작성한 것이 정답이다. LLaMA 계열 모델은 instruction following이 영어에서 더 안정적이다.
- `stripJapanese`를 별도 함수로 분리하고 단계별 주석을 달아 유지보수 가능하게 했다.
- 후처리를 5단계 체이닝으로 명시해 각 단계의 책임이 명확하다.

### 개선 포인트

#### [P1] 상수 선언 위치 역전

현재 파일 구조가 이렇게 되어 있다:

```
const MAX_RECOMMENDATIONS = 10;   // 상수 1
function stripJapanese() { ... }  // 함수
const MAX_TITLE_LENGTH = 100;      // 상수 2 ← 위치가 이상함
const MAX_SUBJECT_LENGTH = 50;     // 상수 3
```

`stripJapanese` 추가 시 편집 위치 때문에 상수가 나뉘었다. `MAX_TITLE_LENGTH`, `MAX_SUBJECT_LENGTH`를 `MAX_RECOMMENDATIONS` 바로 아래로 이동해야 한다.

#### [P2] `daysLeft <= 0 ? "초과"` 분기가 dead code

```typescript
return `${i + 1}. ${a.title} (..., 마감 ${a.daysLeft <= 0 ? "초과" : `${a.daysLeft}일 후`}, ...)`;
```

`getAIComment`는 `recommendations.scheduled`만 받으므로 overdue는 전달되지 않는다. `daysLeft < 0` 케이스는 절대 도달하지 않는다. 단순히 `${a.daysLeft}일 후`로 바꿔도 무방하다.

#### [P3] `stripJapanese`가 중간 문장 일본어 구두점을 제거할 때 문장이 붙을 수 있음

`text。next` → `。` 제거 → `textnext`처럼 공백 없이 붙는 케이스가 이론상 존재한다. 현재 프롬프트 결과에서 발생 가능성은 낮으나, 제거 시 뒤에 공백을 보장하는 방어 코드가 있으면 더 안전하다.

```typescript
.replace(/[\u3001\u3002]/g, " ")  // 일본어 구두점 → 공백으로 대체 후 나머지 정리
```
